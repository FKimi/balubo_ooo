# カスタムコマンド設定ドキュメント

このドキュメントには、開発効率化のために使用しているAIカスタムコマンドのプロンプト設定を記録しています。

---

## `/refactor` カスタムコマンド

### 概要
リファクタリングを定期的に実行することで、コードの長期運用性を向上させます。タスク完了時に毎回実行するレベルで使用しています。

**特徴:**
- リファクタリングが一瞬で完了する
- 精度が高い

### プロンプト

```
# 開発設計ルール

## 原則

- 関心を分離して影響範囲を限定する  

- 高凝集にして内部変更を容易にする  

- 疎結合にして依存関係を減らす  

- 抽象化して変更耐性を高める  

- 非冗長にして重複を排除する  

## 構造

- 同一関心のファイルが3件以上になった場合は専用ディレクトリを作成し階層化する  

## 実装指針

- シンプルを最優先とする  

- オーバーエンジニアリング禁止  

- 常に最小限で設計・実装する  

- ファイルの細分化を避ける。過剰な場合は統合する  

## 運用

- 未コミット変更部分を重点的に確認する  
```

---

## `/commit` カスタムコマンド

### 概要
GUIの✨ボタンでコメント生成していた従来の方法から移行。差分のコメントが日本語になり、ルール付けもできるため、完全にこちらに移行しました。

**特徴:**
- `/commit`で5秒後にはコミットが完了している
- 変更内容を網羅的に確認してからコミット
- 日本語でわかりやすいコミットメッセージを生成

### プロンプト

```
まだコミットしていない変更内容を網羅的に確認して、

どんな修正をしたのかをわかりやすくまとめる。

その後、git add . で変更をステージして、

git commit のコメントとしてその内容を反映する。
```

---

## 使用方法

### `/refactor` の使用タイミング
- **タスク完了時**: 毎回実行してコード品質を維持
- **コードレビュー前**: リファクタリングで整理してからレビュー
- **機能追加後**: 新規コードと既存コードの整合性を確認

### `/commit` の使用タイミング
- **機能実装完了時**: 変更内容をまとめてコミット
- **バグ修正完了時**: 修正内容を明確に記録
- **リファクタリング後**: リファクタリング内容を記録

---
